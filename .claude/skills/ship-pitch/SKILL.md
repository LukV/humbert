# ship-pitch

Move a pitch out of in-cycle. Two paths: **ship** (it landed) and **drop** (overflowed appetite or no longer worth doing). The shipping path also releases via commitizen.

## Vault paths

- Pitch notes (active): `../../docs/planning/pitches/<name>.md`
- Shipped pitches: `../../docs/planning/pitches/shipped/<name>.md`
- In Cycle: `../../docs/planning/In Cycle.md`
- Repo CHANGELOG: `CHANGELOG.md` (auto-generated by `cz bump`)
- User-facing README: `../../README.md`
- Information-manager instructions: `../../docs/technical/001-information-manager-instructions.md`

## Steps for shipping

1. **Read the in-cycle pitch.** From `In Cycle.md`, confirm which pitch is currently in-cycle. Read the pitch note.

2. **Prompt for "what actually happened".** Ask Luk for a one-paragraph summary: what landed, what was cut, what to watch. This is Luk's reflection; the agent's job is to capture it cleanly unless Luk explicitly to write it for him.

3. **Update the pitch frontmatter.** Set `status: shipped`. Add `shipped_on: YYYY-MM-DD` (today). Keep all other fields.

4. **Write the "What actually happened" section.** Replace the placeholder block at the bottom of the pitch (`*(Fill in when the pitch ships...)*`) with Luk's paragraph from step 2.

5. **Move the pitch.** Move `../../docs/planning/pitches/<name>.md` to `../../docs/planning/pitches/shipped/<name>.md` in the vault. Create the `shipped/` directory if it doesn't exist yet.

6. **Clear `In Cycle.md`.** Replace the `## Currently` section body with:

   ```
   ## Currently

   *Nothing in cycle yet.*

   See [[#On deck]] for the next pickup.
   ```

7. **Update the user-facing docs.** A shipped pitch often changes what a user or the information manager can do. Before releasing, check whether the shipped work changed user-facing behaviour (new/changed commands, config, workflow) and, if so, update:
   - **`README.md`** — commands, install/usage, anything a user reads to operate Humbert.
   - **`docs/technical/001-information-manager-instructions.md`** — the IM's hands-on guide: how to author/operate the pack, run evaluation, read refusals and telemetry. Create the file (and `docs/technical/`) if it doesn't exist yet and the shipped pitch gives the IM something to do; otherwise update the relevant section. Skip the file entirely for pitches with nothing IM-facing (e.g. pure frontend scaffolding).

   Keep both in Humbert's tone ([[../../docs/_context.md]]): plain, modest, concrete. If the pitch was purely internal (no user- or IM-facing surface), note that and skip.

8. **Release via commitizen.** Run `cz bump` in the repo root. This bumps the version per conventional-commits, updates `CHANGELOG.md`, and tags. Conventional commits should already have been authored during the cycle via `cz commit`.

   If `cz bump` reports no version bump needed (e.g. only `chore:` commits), report this to Luk and ask whether to skip the bump or force one with `cz bump --increment PATCH`.

9. **Suggest the next pitch.** Read the on-deck list in `In Cycle.md` and propose the next item. Do not auto-`start-pitch` — let Luk confirm.

## Steps for dropping

When a pitch is dropped instead of shipped (overflowed appetite, no longer worth doing):

1. Set `status: dropped`. Add `ended: YYYY-MM-DD`.
2. Write "What actually happened" with explicit emphasis on **why dropped** — overflowed appetite, design changed under it, dependency slipped, no longer valuable.
3. **Do not move the file.** Dropped pitches stay in `pitches/` with `status: dropped` so the dead pitches are part of the record. Future agents reading the planning system should see what was attempted and abandoned.
4. Clear `In Cycle.md` as in step 6 above.
5. **Skip `cz bump`.** Nothing shipped.
6. Suggest the next pitch from on-deck.

## Don't

- Don't bump version on a drop.
- Don't auto-start the next pitch.
- Don't delete dropped pitches — they're part of the planning record.
