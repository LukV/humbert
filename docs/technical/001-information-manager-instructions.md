# Information-manager instructions

This is a hands-on guide for building and running a new domain. Examples of domains are `cheese`, `eumigration`, `lvtm`. The guide instructs you end to end: creating a **data pack**, connecting it, and keeping it good. A pack grows as the product does; evaluation and reading refusals/telemetry arrive with their blocks.

> Who this is for: the **information manager** — they who build and maintain the source that Humbert reads. Asking Humbert questions needs no dbt knowledge; building the source it reads does. This guide assumes none yet — the dbt and MetricFlow primer below is enough to get started, and links out for the rest.

## What a pack is

Humbert does not read a raw database. It reads a **pack**: the governed slice of data it's allowed to answer questions about. The pack is the semantic model — a **dbt project** whose data marts are exposed through a (MetricFlow) semantic layer, with the data in a **DuckDB** file the dbt build produces, plus a `context/` folder for the notes that help you build it.

The split that matters:

- **dbt + your marts** — you own this: the transformations that build the marts,
  and the semantic models/metrics on top of them.
- **The DuckDB warehouse** — generated by `dbt build`; disposable, rebuilt on demand.
- **Humbert** — reads metrics and dimensions through MetricFlow; never the raw tables.

A full pack looks like this — the cheese example (`examples/cheese`) is the worked
reference, so read its `README.md` alongside this guide:

```
mypack/
  dbt_project.yml        the dbt project — the semantic model lives here
  models/
    staging/             clean-up of the raw data
    marts/               the governed public layer (classified `open`)
    semantic/            MetricFlow semantic models + metrics, on top of marts
  context/               design-time notes — Humbert never reads this
  README.md              what the pack holds and how to fill it
  warehouse.duckdb       built by dbt; disposable, rebuilt on demand
```

## A short primer on dbt and MetricFlow

Two tools do the work under a pack. You don't need to be an expert in either to
start, but you do need to know what each is *for*.

**dbt** is the _T_ in E _T_ L. It transforms raw, messy tables into clean, tested ones — the **marts**. You write SQL `SELECT` statements; dbt handles the order they run in, materialises them as tables, and runs tests against them. The convention is to layer it: `staging/` tidies each raw source, `marts/` combines them into the business-shaped tables people actually use. Humbert reads the marts layer and nothing below it.
→ [How we structure a dbt project](https://docs.getdbt.com/best-practices/how-we-structure/1-guide-overview)

**MetricFlow** defines **metrics, measures, and dimensions** on top of the marts,
once, in YAML — so "total production" or "by country" means the same thing every
time, and nobody hand-writes the aggregation. A *measure* is one aggregation over
a mart column; a *metric* is the named, queryable number built on it (step 5
explains how you manage each). It also generates the SQL and works out the joins.
Humbert exposes metrics and asks its questions through MetricFlow; it never
touches the raw tables.
→ [About MetricFlow](https://docs.getdbt.com/docs/build/about-metricflow)

The split, end to end: 

> **raw data → (dbt) marts → (MetricFlow) metrics → Humbert.**

### The commands you'll use

Humbert wraps the important ones for you (`connect` runs `dbt build` then checks
MetricFlow; `vocab` and `query` run MetricFlow). But while authoring it helps to
run them by hand. They need the dbt extra: `uv sync --extra dbt`.

| Command                                      | What it solves                                                                       |
| -------------------------------------------- | ------------------------------------------------------------------------------------ |
| `dbt deps`                                   | Install dbt packages the project depends on.                                         |
| `dbt build`                                  | Run seeds, models, and tests in order — produces the marts and the DuckDB warehouse. |
| `dbt parse`                                  | Generate the manifests Humbert reads, without building data.                         |
| `dbt test`                                   | Run just the tests — your trust check after a change.                                |
| `mf list metrics`                            | See every metric defined.                                                            |
| `mf list dimensions --metrics <name>`        | See what you can group or filter a metric by.                                        |
| `mf validate-configs`                        | Check the semantic models and metrics are well-formed.                               |
| `mf query --metrics <name> --group-by <dim>` | Run a metric and see the numbers.                                                    |

→ Full references: [dbt build](https://docs.getdbt.com/reference/commands/build) ·
[MetricFlow commands](https://docs.getdbt.com/docs/build/metricflow-commands)

## Creating a pack, step by step

### 1. Scaffold the pack folder

```bash
humbert init mypack
```

This creates `mypack/` with a `context/` folder and READMEs. It's idempotent and
never overwrites your edits — safe to re-run. It adds config data to `<HOME>/.humbert/config.json`.

### 2. Add a dbt project

Create the dbt project inside the same folder, using dbt's own tooling
(`dbt init`, or copy the shape of `examples/cheese`). Humbert connects to a
DuckDB-backed project; the cheese example shows a self-contained `profiles.yml`
that keeps the warehouse file in the project folder.
→ [Get started with dbt](https://docs.getdbt.com/docs/get-started-dbt) ·
[dbt-duckdb adapter](https://github.com/duckdb/dbt-duckdb)

**The warehouse path comes from `profiles.yml`.** Humbert reads the DuckDB file's
location from your active output's `path:` (`outputs.<target>.path`) — it does not
assume a filename. A pack scaffolded by `dbt init` writes `dev.duckdb`; the cheese
example writes `warehouse.duckdb`; either works, as long as `profiles.yml` and the
real file agree. If `connect`/`status` report the warehouse as not built when you
know you ran `dbt build`, the usual cause is a `path:` that points elsewhere —
`humbert status` and `/api/healthz` name the file they resolved, so check it
against the file on disk. (Relative paths resolve against the project folder.)

### 3. Build the marts

Model the raw data up into a `marts/` layer — `staging/` to tidy, `marts/` for the
business tables. Keep the layer named `marts` (or tell `connect` otherwise with
`--schema`). Build it:

```bash
cd mypack && dbt build
```

### 4. Classify the governed layer

Humbert exposes a metric **only if every model it reads is classified `open`** —
the **public-only guard**. v0 runs on public data only, and Humbert enforces that
rather than trusting it. Classification rides in dbt `meta:`, set once on the
marts layer in `dbt_project.yml`:

```yaml
models:
  mypack:
    marts:
      +materialized: table
      +schema: marts
      +meta:
        classification: open
```

The guard is **default-deny**: a model with no classification counts as not-open
and is withheld. Forgetting to classify hides data, never leaks it — so when you
add a new mart, you classify it too, or its metrics won't appear. `connect` and
`status` report `K withheld`; `humbert vocab` lists exactly which metrics were
withheld and why. v0 stops at the model level; per-column classification, masking,
and disclosure arrive when non-open data does.

### 5. Define the metrics and dimensions

In `models/semantic/`, write the MetricFlow semantic model (the dimensions and
measures, tied to a mart) and the metrics on top. Trimmed from the cheese pack:

```yaml
semantic_models:
  - name: cheese_production
    description: Cheese production by country, milk source, and year.
    model: ref('fct_cheese_production')
    defaults:
      agg_time_dimension: production_date
    entities:
      - name: cheese_record
        type: primary
        expr: country || '|' || product || '|' || cast(year as varchar)
    dimensions:
      - name: country
        type: categorical
      - name: production_date
        type: time
        type_params:
          time_granularity: day
    measures:
      - name: production_tonnes
        description: Cheese produced, in tonnes.
        agg: sum
      - name: distinct_products
        description: Number of distinct cheese products made.
        agg: count_distinct
        expr: product

metrics:
  - name: total_production
    label: Total cheese production (tonnes)
    type: simple
    type_params:
      measure: production_tonnes
  - name: product_variety
    label: Distinct cheese products
    type: simple
    type_params:
      measure: distinct_products
```

→ [Semantic models](https://docs.getdbt.com/docs/build/semantic-models) ·
[Metrics](https://docs.getdbt.com/docs/build/metrics-overview)

**Measures vs metrics** are easy to confuse, and you manage each differently. A
*measure* is a single aggregation over one mart column — `production_tonnes` sums
the tonnes, `distinct_products` counts distinct products. Measures are the
building blocks: they live in a semantic model's `measures:` list, tied to the
mart they read, and you can't query one on its own. A *metric* is the named number
people actually ask for; the simplest (`type: simple`) just gives one measure a
human `label`, but metrics can also combine measures or other metrics — a ratio, a
running total, a derived sum — so one measure can feed several.

How you manage both:

- **Humbert exposes metrics, not measures.** `humbert vocab` lists metrics, and the
  public-only guard classifies each by the mart(s) its measures read. So every
  number someone should be able to ask for needs a metric — a measure with no
  metric on top is invisible.
- **A measure per aggregation, a metric per question.** Keep measure names plain;
  they're plumbing. Put the wording people see in the metric's `label`.
- One measure with one simple metric on top covers most counts and sums. Reach for
  ratio, cumulative, or derived metric types when a question needs more than a
  single aggregation.

### 6. Fill the context folder

`context/` is for **you**, at design time — Humbert never reads it when answering.
It keeps the knowledge with the pack instead of in someone's head. Park here:

- database dumps and schema exports
- data dictionaries and codebooks
- source notes: where the data comes from, how it's collected, what's missing
- known pitfalls and caveats

This is the material you lean on while modelling the marts and naming metrics.
Keep it close to the work, and update it when the pack changes.

## Connecting and checking

With the pack built, attach it to Humbert and confirm it reads.

```bash
humbert connect ./mypack       # build (if missing) + validate + attach
humbert status                 # confirm it took
```

`connect` builds the DuckDB warehouse if it's missing (force a rebuild with
`--build`), checks MetricFlow can read the metrics, and records the connection.
(`humbert init` on its own just scaffolds `~/.humbert/`; `connect` sets that up
too, so you rarely run it by hand.)

### Exposing the right layer

Humbert only exposes a **governed layer** — by default the dbt schema `marts`.
This is the fence: staging and intermediate models stay out of bounds. Change it
per connection:

```bash
humbert connect ./mypack --schema marts,gold
```

If you rename or move that layer in dbt, `connect` will tell you it found no
models in the exposed schema — re-run with the right `--schema`.

### Reading `humbert status`

```
Connection:  cheese
Project:     /…/examples/cheese   (dbt + DuckDB)
Exposed:     marts   (4 models · 1 metrics · 0 unavailable · 0 withheld)
Warehouse:   /…/examples/cheese/warehouse.duckdb   (built 2026-06-07 14:02)
Skin:        humbert
Locale:      en
```

The **health line** — `N models · M metrics · K unavailable · W withheld` — is
what to watch. `connect` sorts problems into kinds:

- **Fatal** — the project won't parse, or the exposed layer is empty. Nothing
  attaches; fix the dbt project and reconnect.
- **Degraded** — the project attaches, but some metrics are unavailable (e.g. a
  metric points at a column that's gone). `K unavailable` counts them; the rest
  keep working. Treat a rising count as a backlog to fix in the semantic layer.
- **Withheld** — the metric is fine, but its data isn't classified `open`, so the
  guard holds it back. `W withheld` counts them; classify the model to expose it.

### Checking what the semantic layer exposes

Two commands let you inspect — and test — the layer Humbert reads, before any
notebook UI exists. They run through MetricFlow, so they need the `dbt` extra.

```bash
humbert vocab                                  # the metrics + dimensions on offer
```

`vocab` lists each metric and the dimensions you can group or filter it by, with
each dimension's kind (categorical / time) and, for time, its grain. This is the
authored vocabulary — if a metric or dimension you expect is missing, it's a
semantic-layer issue to fix, not a Humbert one.

```bash
# run a selection: a metric, grouped, ordered, limited
humbert query -m total_production \
  --by cheese_record__country --order -total_production --limit 6 --sql
```

`query` composes a selection and validates every name against the vocabulary
*before* it runs — an unknown metric or dimension is reported by name, and nothing
reaches the warehouse until it resolves. Use it to confirm a metric returns the
numbers you expect. Dimension names follow MetricFlow's convention
(`<entity>__<dimension>`, e.g. `cheese_record__country`); `humbert vocab` shows
the exact names. `--where` takes a MetricFlow filter in its **template form** —
`"{{ Dimension('cheese_record__country') }} = 'Germany'"` for a categorical
dimension, `"{{ TimeDimension('metric_time', 'year') }} >= '2015-01-01'"` for a
time one (a raw column name fails unless it's also grouped). `--grain` sets the
time grain, and `--sql` prints the SQL MetricFlow generated — handy for checking
what actually ran.

## Configuring the skin

A **skin** is the branding Humbert wears: its name, colours, and fonts. You set it
with a `theme.json` you write by hand — there's no command for it yet. Place it
beside the connection it brands, at `~/.humbert/projects/<connection>/theme.json`
(or `~/.humbert/theme.json` to brand every connection). With no file, Humbert wears
its default skin and takes `app_name`/`locale` from `config.json`.

The bundled **Proef** skin — CJM brand colours and the Flanders Art fonts — is the
worked example:

```json
{
  "app_name": "Proef",
  "locale": "nl",
  "colors": {
    "primary": "#2B979D",
    "secondary": "#CC5621",
    "accent": "#5D6009",
    "palette": ["#2B979D", "#CC5621", "#5D6009", "#1A6B70", "#E8895A", "#8A8C2E"]
  },
  "fonts": {
    "body": "Flanders Art Sans",
    "editorial": "Flanders Art Serif",
    "custom_css": "/fonts/flanders-fonts.css"
  }
}
```

- **`app_name`** — the name shown in the interface.
- **`locale`** — the interface language, `en` or `nl`.
- **`colors`** — `primary` drives the accent and the logo; `palette` sets the chart
  category colours, in order. `secondary` and `accent` fill out the palette when you
  don't list one. In dark mode your brand colours are used as you give them, so pick
  ones that stay legible on a dark background.
- **`fonts`** — `body` and `editorial` name font families; `custom_css` is a
  stylesheet the app loads to supply them. The Flanders Art faces ship with Humbert
  (served at `/fonts/…`), so the example above works as-is. For other fonts, add the
  files under `apps/web/public/fonts/` and rebuild the frontend.

Humbert reads `theme.json` on every load, so a **browser reload** picks up your
edits — no restart needed (new font *files* do need a frontend rebuild). `humbert
status` shows the skin name and locale recorded in `config.json`.

> One small gap today: `logo_path` isn't drawn as an image (the logo is tinted by
> `primary` instead). The light/dark switch in the corner is a separate, per-person
> browser preference — not part of the skin.

## Updating a pack

Authoring isn't one-and-done. The common changes and what each needs:

- **New metric or dimension** — add it in `models/semantic/`, then `dbt build` and
  `humbert connect ./mypack --build` to re-validate.
- **New mart** — classify it (`classification: open`) or it stays withheld, then
  rebuild and reconnect.
- **Refreshed underlying data** — `humbert connect ./mypack --build` rebuilds the
  warehouse from the new data.
- **Renamed the governed layer** — reconnect with `--schema <new-name>`.

The rule of thumb: change the dbt or semantic files → rebuild → reconnect.

## Best practices

- **Keep metrics on the marts layer.** That's what Humbert treats as the source of
  truth; metrics over staging or intermediate models invite drift.
- **Classify as you go.** Add `classification: open` the moment you add a mart, so
  default-deny never silently hides something you meant to expose.
- **Name for the person asking, not the table.** Metric labels and dimension names
  are what questions map onto — make them read like plain language.
- **Test the marts.** dbt tests are your trust check; a pack that builds green is
  one you can stand behind.
- **Write the context down.** A data dictionary in `context/` today saves an hour
  of remembering in six months — and lets someone else maintain the pack.
- **Verify with `vocab` and `query` before you trust it.** Run a metric by hand and
  check the numbers are what you expect.
- **Start small.** One clean metric people trust beats ten nobody checked.
